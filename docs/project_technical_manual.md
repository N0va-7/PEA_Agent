## 2. 项目定位

PEA Agent 是一个面向企业邮件安全场景的钓鱼邮件分析系统。系统以 `.eml` 为输入，输出结构化判定结果与中文分析报告，并通过人工反馈闭环实现持续优化。

核心目标：

1. 提升钓鱼邮件识别的准确性与可解释性。
2. 保证结果可追溯、可复核、可运营。
3. 在不引入高风险自动漂移的前提下，实现参数持续迭代。

## 4. 项目创新点

### 4.1 多节点可解释分析工作流

系统并非单点模型推断，而是按“解析 -> 特征提取 -> 多维分析 -> 融合决策 -> 报告”编排流程，具备以下优势：

1. 可定位误判来源（正文、URL、附件或融合阈值问题）。
2. 可替换局部节点而不重构全链路。
3. 可输出阶段进度，便于运营监控。

### 4.2 动态置信融合决策

在 URL 与正文概率融合时，不采用固定权重，而是引入置信修正：

1. 置信度定义：`c = |p - 0.5| + 0.5`。
2. 置信高的信号在同一封邮件中获得更高融合权重。
3. 减少单一低置信模型导致的误判放大。

### 4.3 人工反馈闭环 + 手动激活机制

系统支持“标注 -> 调参 -> 激活 -> 回滚”全生命周期管理，核心创新在于：

1. 调参必须手动触发，避免小样本自动漂移。
2. 调参前强制 Precheck 门槛校验，失败即拒绝运行。
3. 参数版本化入库，只有显式激活才影响线上判定。

### 4.4 稳定中文报告生成机制

1. 后端使用固定章节模板生成中文报告。
2. LLM 仅用于结构化补全，不作为唯一输出来源。
3. LLM 失败或返回非中文时自动回退默认中文内容。
4. 前端渲染失败时回退为纯文本预览，保证可读性。

## 5. 系统架构与代码骨架

### 5.1 分层架构

1. `frontend/`：交互层（上传、历史、反馈、调参管理）。
2. `backend/api/`：接口层（认证、分析、报告、调参、系统信息）。
3. `backend/services/`：应用服务层（任务处理、流程编排调用）。
4. `backend/workflow/`：分析图层（节点与路由）。
5. `backend/repositories/`：数据访问层。
6. `backend/models/`：数据库实体。
7. `ml/training/`：离线训练与调参脚本。
8. `ml/artifacts/`：线上模型与调参产物。

### 5.2 后端关键模块职责

1. `backend/main.py`：应用启动、依赖装配、全局异常处理、队列启动。
2. `backend/services/job_runner.py`：任务执行器，支持 memory/redis 两种后端。
3. `backend/services/analysis_service.py`：分析任务生命周期管理与进度事件记录。
4. `backend/workflow/graph.py`：LangGraph 工作流构建。
5. `backend/workflow/nodes/*.py`：各节点实现（解析、模型分析、融合、报告、持久化）。

### 5.3 数据库模型骨架

1. `email_analyses`：分析主表（分析结果、报告路径、反馈最新值）。
2. `analysis_jobs`：任务状态与进度事件。
3. `analysis_feedback_events`：反馈变更审计日志。
4. `fusion_tuning_runs`：调参运行历史与指标。
5. `system_config`：系统级激活参数指针（如当前生效调参版本）。

## 6. 核心业务流程规划

### 6.1 邮件分析主流程

1. 前端上传 `.eml`，后端创建 `job_id` 并入队。
2. 工作流计算邮件指纹并查重。
3. 命中历史则直接返回缓存结果。
4. 未命中则执行解析、URL提取、附件分析、正文分析、URL分析。
5. 综合判定节点输出 `is_malicious + score + reason`。
6. 报告节点生成结构化中文 Markdown。
7. 持久化节点写库并保存报告文件。
8. 前端通过 `jobs/{job_id}` 轮询进度并展示结果。

### 6.2 人工反馈流程

1. 分析员在历史详情页标注 `malicious` 或 `benign`。
2. 系统更新主表最新反馈字段。
3. 系统追加审计事件到 `analysis_feedback_events`。

### 6.3 融合调参流程

1. 管理员先执行 `precheck`。
2. 系统验证样本量、类别分布、近期反馈活跃度。
3. 通过后由管理员手动确认运行 `run`。
4. 系统网格搜索并保存最优参数与 Top-K 候选。
5. 管理员手动 `activate` 指定版本。
6. 决策节点优先读取激活版本参数。

### 6.4 运维观察流程

1. 总览页展示数据库/队列连接信息（脱敏）。
2. 历史页支持筛选、删除、清空和报告下载。
3. 任务阶段事件可追踪处理链路状态。

## 7. 模型训练与调参方法

### 7.1 训练数据与标签

1. 正文训练集：`ml/training/email_text/Phishing_Email.csv`。
2. URL训练集：`ml/training/email_url/phishing_site_urls.csv`。
3. 调参数据集：来自人工反馈样本，映射为 `url_prob,text_prob,label`。
4. 训练目录仅保留脚本与数据源，不再保留旧版 `model/*.pkl` 副本。

### 7.2 训练流程

1. 分层切分 `train/val/test = 70/15/15`。
2. 正文模型与 URL 模型独立训练。
3. 模型采用可输出概率的线性分类器（`SGDClassifier(loss=log_loss)`）。
4. 在验证集按 FPR 约束选阈值。
5. 在测试集做一次最终评估并固化报告。

### 7.3 融合方法

基础融合公式：

```text
score = w_u * p_url + w_t * p_text
```

动态置信修正：

```text
c_u = |p_url - 0.5| + 0.5
c_t = |p_text - 0.5| + 0.5
w_u = (w_url_base * c_u) / (w_url_base * c_u + w_text_base * c_t)
w_t = 1 - w_u
```

### 7.4 调参策略

1. 搜索空间：
   - `w_url_base: 0.0 -> 1.0`
   - `threshold: 0.50 -> 0.95`
2. 先筛选 `FPR <= target` 候选。
3. 在满足 FPR 约束的候选中优先 Recall，再参考 F1。
4. 若无候选满足 FPR，退化选择最小 FPR 方案。

### 7.5 线上参数读取优先级

1. 当前激活调参版本（`system_config` 指针）。
2. `ml/artifacts/fusion_tuning*.json` 最新文件。
3. 默认参数（代码内置）。

线上模型主产物：

1. `ml/artifacts/phishing_body.pkl`
2. `ml/artifacts/phishing_url.pkl`

## 8. 安全与可靠性设计

### 8.1 安全机制

1. JWT 鉴权保护业务接口。
2. 登录接口按用户名+来源地址限流，防止暴力尝试。
3. 密码哈希支持 PBKDF2，并兼容历史 SHA256 存量。
4. 后端错误返回默认脱敏，避免泄漏内部细节。

### 8.2 可靠性机制

1. 队列 worker 捕获异常并保持存活。
2. 调参运行限制单并发，避免参数竞争写入。
3. 报告路径校验限制在根目录内，防止非法路径访问。
4. 历史删除时同步删除关联报告与反馈审计。

### 8.3 可追溯性机制

1. 任务节点级事件日志（开始/完成/失败）。
2. 分析结果包含 `execution_trace`。
3. 调参运行包含触发人、时间、样本量、最优参数、候选集。
4. 反馈修改保留完整审计记录。

1. `POST /api/v1/auth/login`：登录并签发令牌。

### 9.2 分析与历史

1. `POST /api/v1/analyses`：上传 EML 并创建任务。
2. `GET /api/v1/jobs/{job_id}`：轮询任务状态与进度。
3. `GET /api/v1/analyses`：历史列表检索。
4. `GET /api/v1/analyses/{analysis_id}`：读取分析详情。
5. `DELETE /api/v1/analyses/{analysis_id}`：删除单条历史。
6. `DELETE /api/v1/analyses`：清空历史。

### 9.3 报告与反馈

1. `GET /api/v1/reports/{analysis_id}`：下载报告。
2. `POST /api/v1/analyses/{analysis_id}/feedback`：写入反馈。
3. `GET /api/v1/analyses/{analysis_id}/feedback-history`：读取反馈审计。

### 9.4 调参与系统信息

1. `POST /api/v1/tuning/fusion/precheck`：调参前门槛检查。
2. `POST /api/v1/tuning/fusion/run`：手动运行调参。
3. `GET /api/v1/tuning/fusion/runs`：调参历史。
4. `POST /api/v1/tuning/fusion/runs/{run_id}/activate`：激活版本。
5. `GET /api/v1/system/runtime-info`：读取连接信息（脱敏）。

## 13. 结论

PEA Agent 的技术核心不是单一模型，而是“可解释分析流程 + 受控反馈调参 + 参数版本治理”的工程化体系。该体系在毕设场景下具备明确创新点、可复现实验路径和可落地部署方案，可直接支撑论文的“系统设计、实现与实验分析”章节。
